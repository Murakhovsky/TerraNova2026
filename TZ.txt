# Terra Nova CLUB

## ТЗ для фронтенду: зведена основа v0.3

## 1. Суть продукту

**Terra Nova CLUB** — головний публічний портал екосистеми навколо нерухомості, девелопменту, технологій, інвестицій та арт-напряму.

Портал має одночасно працювати як:

* преміальна презентація бренду;
* каталог власних і партнерських об’єктів;
* точка входу в Listing / MLS;
* частина воронки продажів;
* точка входу в CRM та AIDA;
* вітрина для інвесторів;
* 3D-простір для презентації об’єктів і концепцій;
* платформа для майбутнього масштабування в Україні та за кордоном.

Основне враження від порталу:

> Тут є цікава нерухомість, сучасні технології, сильні ідеї та можливості для інвестицій.

## 2. Архітектура брендів

### Terra Nova CLUB

Головний портал, головна бренд-оболонка, презентація екосистеми.

### Terra Nova Agency

Напрям продажу, оренди, інвестиційної нерухомості, роботи з рієлторами та партнерами.

До складу Terra Nova Agency входять:

* каталог об’єктів;
* Listing / MLS;
* робота з власниками;
* робота з рієлторами;
* заявки;
* продажі;
* комісії;
* добірки об’єктів.

### Terra Nova D

Девелопмент, власні ЖК, котеджі, земельні ділянки, будівництво, інвестиційні проєкти.

### Terra Nova Art

Презентаційний напрям першого релізу:

* мистецтво;
* інтер’єри;
* арт-об’єкти;
* візуальна культура;
* інтеграція мистецтва в простір і нерухомість.

### Terra Nova IT

Окрема сторінка та технологічний напрям:

* EstateBook CRM;
* AIDA;
* AI-інструменти;
* автоматизація;
* n8n;
* GPT;
* майбутня токенізація.

### EstateBook

Внутрішній CRM-продукт, який має бути видимим у меню Terra Nova CLUB.

EstateBook не позиціонується як окремий публічний головний бренд, але має бути технологічним ядром екосистеми.

## 8. 3D-модуль

### Технологічна основа

3D-модулі створюються на базі **Three.js** як окремі фронтенд-модулі, що підвантажуються лише на потрібних сторінках. Three.js підтримує базові 3D-сцени, камери, взаємодію та модульну структуру для браузерних застосунків.

### Типи 3D-сцен

1. Віртуальний тур по конкретному об’єкту.
2. Інтерактивний генплан.
3. Віртуальна галерея Terra Nova Art.
4. 3D-модель будинку або ЖК.
5. Простір із клікабельними точками інтересу.

### Вимоги

* до 5 типових 3D-сцен;
* навігація від першої особи;
* аватари;
* клікабельні об’єкти;
* точки інтересу;
* інтеграція карток нерухомості у простір;
* адаптація для mobile;
* fallback-режим для слабких пристроїв;
* можливість VR-перегляду;
* без окремої аналітики 3D-взаємодій у Release 0.

### Ризик

Готових 3D-моделей немає і відповідальний за їх виробництво не визначений. Це блокує створення реальних 3D-сцен, навіть якщо код написаний бездоганно, що, як відомо, трапляється приблизно раз на геологічну епоху.

## 14. Технічна архітектура

### Бекенд

* PHP;
* Phalcon;
* REST API;
* частина endpoint-ів уже існує;
* API потрібно провести через аудит;
* endpoint-и потрібно дописати;
* API-документацію готує GPT, але її потрібно вручну перевірити перед використанням фронтендом.

### Фронтенд

Прийнятий підхід:

* фронтенд не є окремим SPA-застосунком;
* рендеринг відбувається через шаблонізатор Phalcon;
* каталог і SEO-сторінки рендеряться сервером;
* інтерактивні модулі підключаються окремими JavaScript-бандлами;
* Three.js працює як ізольований модуль;
* CRM і AIDA адаптуються з існуючих версій;
* REST API використовується для динамічних даних.

Якщо використовується Volt, він підходить для серверного рендерингу в Phalcon і повторного використання шаблонних компонентів та partials.

* local;
* dev;
* staging;
* production;
* автоматичний deploy;
* GitHub Actions;
* окремі environment variables та secrets;
* окремі доступи для staging і production.

GitHub Actions підтримує окремі deployment environments, secrets і правила розгортання для staging та production.

## 6. Авторизація та ролі

### Основна схема

Firebase Authentication є єдиною системою ідентифікації користувача.

Після авторизації:

1. Firebase виконує вхід користувача.
2. Фронтенд передає Firebase ID token у Phalcon через `Authorization: Bearer`.
3. Phalcon перевіряє токен.
4. Phalcon знаходить або створює локальний профіль користувача.
5. Ролі, права доступу, зв’язки з об’єктами та CRM-даними зберігаються у вашій БД.
6. Новий окремий механізм login/session у Phalcon не створюється.

### Способи входу

* email + пароль;
* email-підтвердження;
* Google Login;
* Apple Login;
* Telegram Login;
* двофакторна перевірка через email або Telegram.

Firebase документує готові потоки для email/password, Google та Apple, а для нестандартної ідентифікації підтримує custom-auth flow через власні токени. У ТЗ Telegram Login слід реалізувати саме як кастомний серверний сценарій: Phalcon перевіряє Telegram-авторизацію, після чого видається Firebase custom token.

### Ролі

* покупець;
* рієлтор;
* власник;
* інвестор;
* адміністратор.

Один обліковий запис може мати кілька ролей.

Відео на першому екрані не має бути обов’язковою частиною завантаження. Воно завантажується після першого рендеру або замінюється статичним постером на mobile.

* після завантаження система автоматично створює:

  * WebP;
  * AVIF, якщо підтримується;
  * thumbnail;

На сайті не дозволяється показувати оригінали фото, якщо є оптимізовані варіанти.

* приватні документи доступні лише авторизованим ролям із відповідними правами;
* PDF не повинен лежати всередині Git-репозиторію.

### Відео

* пряме завантаження великих відео на сервер у Release 0 не рекомендується;
* відеоогляди підключаються через YouTube, Vimeo або інший зовнішній відеохостинг;
* максимальна вага короткого локального відео: до 30 MB;
* формат: MP4 H.264;
* відео завантажується ліниво після взаємодії користувача або після рендеру основної сторінки.

## 11. Критерії готовності Release 0

Release 0 вважається готовим, коли:

1. `terra-nova.club` відкривається без критичних помилок.
2. Головна сторінка, каталог, картка об’єкта й ключові презентаційні сторінки доступні на mobile та desktop.
3. У каталозі є щонайменше 30 опублікованих об’єктів.
4. Фільтрація, карта, вибране та порівняння працюють.
5. Кожна форма створює лід у CRM.
6. Відповідальний менеджер отримує Telegram-сповіщення.
7. AIDA доступна як чат на всіх публічних сторінках.
8. Авторизація через Firebase працює.
9. Користувач може увійти через email, Google, Apple або Telegram.
10. EstateBook CRM доступна на окремому субдомені.
11. Listing / MLS має публічну вітрину та закритий робочий режим.
12. Усі ключові сторінки мають title, description, Open Graph, canonical та sitemap.
13. Метрики GA4, GTM, Meta Pixel та ключові конверсії збираються.
14. Сайт не має критичних помилок у Chrome, Safari, Firefox та Edge останніх двох версій.
15. Основні сторінки відповідають технічним межам продуктивності цього документа.
